# InterModule — Roadmap к «портируемому максимуму скорости» + OSS-библиотека

North star: **своя fp16-компьют-реализация на wgpu (один код native + браузер) + минимальная
модель + GPU-resident пайплайн**, за `trait FrameInterpolator` с быстрым TensorRT-бэкендом на
NVIDIA. Цель — real-time интерполяция, которую можно встроить (в т.ч. в браузер) и отдать в OSS.

Где мы сейчас (2026-07-01):
- ✅ Native TensorRT in-process из Rust: 720p 20.8мс (48fps PASS), сквозной pipeline без Python.
- ✅ Браузер доказан: RIFE-ONNX бежит на WebGPU (ort-web), корректно — но **1712мс/кадр**
  (GridSample + shape-ops падают на CPU).
- Узкое место всего проекта: bandwidth-bound модель → **главный рычаг = модель, не рантайм**.

Легенда: 💰 = прирост на вложенный час (ROI). Усилия: ч=часы, д=дни, н=недели, м=месяцы.

## Фаза 0 — Фундамент (1–2 дня) — делать ПЕРВЫМ

Без этого каждый новый бэкенд = копипаст. Дёшево, разблокирует всё.

- [ ] `crates/rife-core`: `trait FrameInterpolator { fn interpolate(&self, f0:&Frame, f1:&Frame)->Result<Frame> }`, тип `Frame`.
- [ ] **Единый pre/post** (RGB↔BGR, /255, pad/crop) в одном модуле — убрать 4 копии (Python, candle `lib.rs`, `trt.rs`).
- [ ] Backend-agnostic ffmpeg-пайплайн (убрать дубль `read_exact_or_eof`).
- [ ] Импл `RifeTrt` и `RifeCandle` (последний помечен «reference/oracle») за trait.
- [ ] Воспроизводимая сборка: `scripts/bootstrap_trt` (регенерация import-lib + fetch headers), автодискавери CUDA в `build.rs` (не хардкод v13.1).

**Done when:** `cargo test` гоняет один и тот же кадр через оба бэкенда и сравнивает; сборка на чистой машине по одному скрипту.

## Фаза 3 — Свой WGSL-warp (1–2 недели) 🔥

> ⚠️ **ЗАМЕРЕНО НА 4060 Ti (2026-07-02): свой warp сам по себе НЕ спасёт.** ort-web WebGPU =
> **~1900 мс/кадр (0.5 fps)**, fp16 не помогает. GridSample УЖЕ на GPU (`force CPU node count:0`),
> а тормозит окружение: shape/Slice-операции на CPU → GPU↔CPU синки + общий per-op overhead
> ort-web. Native TRT тот же кадр = ~25мс → браузер **~75× медленнее**, и это оверхед рантайма,
> не модель. Свой WGSL-warp уберёт стоимость GridSample (которой уже нет), но не синки/оверхед.
> Реальные рычаги: **свой wgpu-бэкенд целиком (Фаза 6)**, **шринк модели (Фаза 5)**, либо
> **принять браузер как offline/slow-mo-only**. См. `docs/phase2_ortweb_ceiling.md`.

Крупнейший браузерный скачок. Аналог твоего CUDA-warp, но на WebGPU.

- [ ] Реализовать backward-warp (bilinear, border) как WGSL compute-шейдер.
- [ ] Встроить как кастомную операцию (ort-web custom op) ИЛИ вынести warp из графа и делать его в своём wgpu-проходе между сессиями.
- [ ] Убрать 14 GPU↔CPU roundtrip'ов.

**Done when:** ноль CPU-fallback на горячем пути; ожидаемо 5–20× к браузерной латентности.

## Фаза 4 — GPU-resident пайплайн (1–2 недели)

Часто важнее, чем оптимизация самих кернелов — скорость теряется в трансферах.

- [ ] WebCodecs `VideoDecoder` → GPU-текстура напрямую (без CPU-копий).
- [ ] Инференс → результат остаётся в GPU-текстуре → отрисовка.
- [ ] Overlap: декод(N+1) ∥ инференс(N) ∥ отрисовка(N-1).
- [ ] fp16 по всему пути.

**Done when:** ни одного лишнего CPU↔GPU копирования на кадр; стабильный fps.

## Принципы

- **Всегда иметь рабочий demo-able артефакт** после каждой фазы (не ломать существующее).
- **Модель — главный рычаг** (bandwidth-bound). Оптимизация исполнения имеет жёсткий потолок.
- **Не гнаться за одним «идеалом»**: TensorRT = абсолютный максимум на NVIDIA; wgpu = максимум портируемый. Держать оба за `trait`.
- **Трансферы дороже матмулов** — GPU-resident пайплайн часто даёт больше, чем микро-опт кернелов.
