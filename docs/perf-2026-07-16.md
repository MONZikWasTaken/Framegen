**Date:** 2026-07-16 · **Hardware:** RTX 4060 Ti (Ada, 34 SM, 288 GB/s), Windows 11,
Chrome/Dawn D3D12 · **Method:** every lever A/B-benched in-session via

## Landed (commits aaacace..d516a3b)

1. **conv0b stride-2 sweep in the tuner** (`aaacace`): wgslConvRBs2 gained the w4
   treatment (5x5 window in f32 registers once per ci, pre-widened f32 weights -
   the quarter-rate f16->f32 CVTs leave the hot loop) and the tuner now benches
   {coc 8/16} x {w4} at the real conv0b shape, shipping the winner as
   `convTune.s2`. conv0b isolated @1080p: 0.92 -> 0.39 ms (w4+coc16). Bit-exact.
2. **coc16 cb variants in the tuner** (`aaacace`): halve the z-slices re-staging
   the same tile; won at 1080p (0.41 -> 0.35 isolated), tuner keeps coc8 at
   720p/480p where it still wins. Bit-exact (accumulation order unchanged).
3. **fcTune3** (`e02b235`): calibrateConvTune persisted only {coc,slab,sg,wgx,wgy}
   and silently DROPPED the tuner's w4/v2 flags - every calibrated extension user
   ran the legacy kernels (~+30% mid time) and SR fell back the same way (~2x).
   The full tune (plus s2) now persists; the storage key bump forces one
   recalibration. The worker player had the same bug (`36113c1`, framegenTune2).
4. **SR per-size conv self-tune** (`3a1f9cc`): the SR convs (16ch at FULL video
   res) ran the trunk's tune (96-240ch at H/16). The async per-size state build
   now benches the few candidates on the real size: mid coc16 (one z-slice),
   out conv at z=1 (coc=C4: 1.38 -> 0.91 ms isolated). Byte parity max|d|=0.
5. **SR in-conv via textureLoad, 16x8 wg** (`d516a3b`): the linear sampler at
   texel centers IS the exact texel; textureLoad drops the sampler/uv work
   (0.47 -> 0.37 ms isolated). Parity max|d|=0.
6. **Player worker JIT mids** (`36113c1`): runMulti batched all mids into one
   submit - the first mid waited for the whole batch (~(n-1)*ms extra latency at
   high factors). Ported the extension's prepPair + per-mid runT scheme driven
   from the pump, plus the non-blocking 3-slot dedup readback ring (handleFrame

9. **Native + tools** (`10da854`): warp.rs compiled its CUDA kernel with NVRTC on

## Tried and KILLED this session (all measured, same harness)

- **Two-phase tuner prune** (short pass -> finalists): 8-dispatch rankings are
  noisy enough to drop the true winner - measured a wrong pick costing +20% on
  the 2x cycle. Tuner correctness beats burst length; reverted.
- **wv4** (vec4 weight rows 9->12, plain uniform loads instead of
  subgroupBroadcastFirst): ~30% SLOWER isolated (0.48 vs 0.37) - the broadcast
  hint genuinely helps on Ada/DXIL. Removed.
- **t32** (tile staged as vec2<f32>, window CVTs out of the loop): exact tie with
  the best f16 shapes (0.35 vs 0.35-0.37) but +30% tuner burst and noisy pick
  flip-flops between tied variants (one run regressed -2.4%). Removed.
- **RCOC=16 for the refine chain** (z=1 like the SR out conv): mid @1080p motion
  1.86 -> 1.93 ms. Reverted.
- **SR px-per-thread kernels** (whole 9KB weight tensor in shared, no slabs/z):
  1.9 ms vs RB's 1.1 - weight staging amortizes over only 64 px/wg (14400 wgs).
  The fused out+shuffle variant of the same idea: 1.47 vs 1.16 split. Not adopted.
- **SR texture-activation conv** (16ch in 4 rgba16float textures, no shared, no
  barriers, CVT in the texture unit): 1.77 vs 1.1. The texture path does not
  beat shared-tile staging at c16 full-res on Ada. Not adopted.
- **Winograd F(2x2,3x3) for the cb stack** - killed by arithmetic before
  implementation: the premise ("cb is at the FMA floor, only MAC reduction
  helps") holds for the FULL model (C2=240: floor 0.169ms vs ~0.19 measured) but
  NOT for the shipping v7s (C2=96: floor 0.061ms vs 0.353 measured = 17% of
  peak, issue/latency-bound). Winograd cuts FMAs that are not the limiter, and
  its 16 f32 accumulators per (thread, co) force either COC=4 (4x tile
  re-staging, the exact cost that made coc4 lose 3x to coc16) or 128+ live
  registers (the coc24 probe with 96 accumulators already regressed 0.35 ->
  0.64). Both doors closed empirically.

The instruction-diet path on the cb kernel is exhausted at ~0.35ms/cb @1080p on
this GPU (~17% of FMA peak, plateau shared with the SR c16 convs). The next
step-change remains subgroup-matrix (tensor cores), tracked separately.
